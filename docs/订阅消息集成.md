# 订阅消息集成

> 把应用内 `family_inbox` 的通知升级为**触达用户手机微信通知栏**的订阅消息。
>
> 架构已就位 — 代码链路、云函数、前端授权、触发点全部完成。**缺的只是 3 个模板 ID**。

## 整体链路

```
【业务事件】
   ├ anomalyDetector.detectAndPush       ──┐
   ├ replenishDispatcher.dispatchToChild  ──┤
   └ digestService.generateForMe          ──┤
                                            │
                                            ▼
                          subscribeMessageService.send(tmplKey, openid, data, page)
                                            │
                                            │ wx.cloud.callFunction('notify')
                                            ▼
                        云函数 cloudfunctions/notify/index.js
                                            │
                                            │ cloud.openapi.subscribeMessage.send
                                            ▼
                               微信服务端 → 用户手机通知栏
```

## 申请 tmplId（必须人工做）

### 前置条件
- 小程序已认证为**企业主体**（个人主体不能开通订阅消息）
- 小程序类目已包含"健康类"或"电商-综合电商"（对应模板库的行业）

### 步骤

1. 登录 **https://mp.weixin.qq.com** 小程序后台
2. 进入【功能】→【订阅消息】
3. 若未开通，先申请开通（需要主体资质匹配）
4. 【我的模板】→【添加模板】→ 从公共模板库选择 / 自定义

**建议模板**（各一条）：

#### 模板 1：健康异常预警（`healthAnomaly`）
- 行业：健康（或公共模板库搜"健康提醒"）
- 场景：父母的健康数据出现异常时通知子女
- 建议字段：
  ```
  thing1   异常内容      示例："静息心率 110 bpm"
  thing2   建议          示例："建议尽快查看父母状态"
  time3    时间          示例："2026-04-22"
  ```

#### 模板 2：补货提醒（`replenishDue`）
- 行业：电商-综合电商
- 场景：父母常用的保健品消耗殆尽时通知子女代购
- 建议字段：
  ```
  thing1   商品名        示例："宁夏中宁特级枸杞"
  thing2   提醒类别      示例："已到补货期" 或 "即将用完"
  time3    预计时间      示例："2026-04-22"
  ```

#### 模板 3：周健康简报（`weeklyDigest`）
- 行业：健康
- 场景：每周一次的父母健康简报生成后通知子女
- 建议字段：
  ```
  thing1   简报标题      示例："父母这周睡眠稳定"
  thing2   建议内容      示例："点开可看 AI 关心与建议"
  time3    起始周        示例："2026-04-22"
  ```

> **字段名必须与代码一致**：当前代码按 `thing1/thing2/time3` 组装 data。若模板用了别的字段名（如 `name1/phrase2/date3`），需要同步修改下列文件里的 data 对象：
> - `miniprogram/services/anomalyDetector.js` 的 `subscribeMessageService.send` 调用
> - `miniprogram/services/replenishDispatcher.js` 同上
> - `miniprogram/services/digestService.js` 同上

### 审核时长
通常数小时到 1-2 个工作日。通过后在【我的模板】看到每个模板的 tmplId（形如 `_vFODEIDw-xxxxxxxxxxxxxxxxxxxxxxxxx`）。

## 填入 tmplId

打开 `miniprogram/config/index.js`，把：

```js
subscribeTemplates: {
  healthAnomaly: "REPLACE_ME_health_anomaly_tmpl",
  replenishDue:  "REPLACE_ME_replenish_due_tmpl",
  weeklyDigest:  "REPLACE_ME_weekly_digest_tmpl"
}
```

替换为真实值：

```js
subscribeTemplates: {
  healthAnomaly: "_vFODEIDw-xxxxxxxx",
  replenishDue:  "_vFODEIDw-yyyyyyyy",
  weeklyDigest:  "_vFODEIDw-zzzzzzzz"
}
```

保存 → 小程序开发者工具**重新编译** → 流程立即生效。

## 调试

### 1. 验证前端授权弹窗
- 子女端进入 `pages/family-home`，看到"🔔 开启微信推送"banner
- 点击 → 应弹出微信原生订阅对话框（**只有在真 tmplId 生效时才弹**）
- 若仍是 placeholder，subscribeMessageService 会跳过 wx.requestSubscribeMessage，显示"微信推送未就绪"弹窗

### 2. 验证云函数调用
- 在老人端触发业务事件（例如 digest 页点"生成本周简报"）
- 在微信开发者工具的【云开发】面板 → 【云函数】→ `notify` → 【日志】
- 看到调用记录；若是 placeholder 会返回 `{success: false, errCode: "PLACEHOLDER_TMPL_ID"}`
- 若 tmplId 正常但 data 字段对不上模板，会返回 `{errCode: 43101, errMsg: "用户...subscribe..."}` 等

### 3. 真实触达
- 子女在真机上授权后，老人端触发生成简报
- 子女微信应收到服务通知（在【订阅消息】里）
- 点击可跳转到配置的 `page`（如 `pages/digest-detail/index?id=xxx`）

## 前置风险

| 错误 | 原因 | 处理 |
|---|---|---|
| `errCode: 43101` | 用户未 acceptSet / 已过期 | 前端在相关操作时重新 requestSubscribeMessage |
| `errCode: 47003` | 模板参数格式不对 | 对照模板定义改 `data` 字段 |
| `errCode: 41030` | 页面路径不在 pages 列表 | 确保 `page` 参数指向合法路径 |
| placeholder 无弹窗 | tmplId 未替换 | 按上述步骤替换 |
| 订阅消息未开通 | 主体不符 / 未申请 | 先完成主体认证 + 能力申请 |

## 架构设计原因

**为什么用云函数而不是 CloudRun？**
- `openapi.subscribeMessage.send` 在 `wx-server-sdk` 里原生支持，云函数直接用，零配置
- 单职责：只做消息发送，和业务逻辑解耦

**为什么失败不阻塞业务？**
- 订阅消息是**附加通道**，in-app inbox 是主通道
- 模板审核/用户未授权/openapi 限流等失败情况频繁，不能让业务挂

**为什么 placeholder 要在云函数里也拦截？**
- 双重防御：前端 `isPlaceholder` 不调 `wx.requestSubscribeMessage`；即使前端变更跳过这层，云函数拒绝向微信 openapi 发起请求，避免 appid 在微信侧记"调用异常"
