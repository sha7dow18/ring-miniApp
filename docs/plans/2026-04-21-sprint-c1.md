# Sprint C1 — 数据模型与角色切换骨架

## 目标

为 PRD 三大核心功能（体质辨识 / AIGC 内容 / AI 电商运营双端）铺好底座：给用户画像加 **role / constitution / boundFamilyId** 三个字段、建家庭绑定与 inbox 两个集合、新增角色选择与绑定两个页面。本身不实现任何业务闭环。

## 范围

### 云端
- 新建集合 `family_bindings`：`elderOpenId`、`childOpenId`、`inviteCode`、`status`、`createdAt`、`boundAt`
- 新建集合 `family_inbox`：`toOpenId`、`fromOpenId`、`type`、`title`、`body`、`payload`、`read`、`createdAt`
- `user_profile` 字段扩展靠前端写入生效（NoSQL schemaless）

### 代码
1. `services/profileService.js`：`DEFAULT_PROFILE` 加 `role: null` / `constitution: null` / `boundFamilyId: null`；去掉无意义的 `.catch(() => null)`（按新指令简洁优先）
2. `services/familyService.js`（新）：`generateInviteCode()`（6 位大写字母+数字）、`createPendingBinding(role)`、`redeemInviteCode(code)`、`getMyBinding()`、`unbind()`
3. `services/familyInboxService.js`（新）：`listInbox(limit)`、`markRead(id)`、`pushToInbox(payload)` —— 生产端到 C6 才调
4. `app.js`：`onLaunch` 拉 profile.role 写入 `globalData.role`；空时跳 `/pages/role-switch/index`
5. `app.json`：注册两个新页面
6. `pages/role-switch/`：全屏双按钮 + 选定后写 role、`reLaunch` 到 `pages/home/index`
7. `pages/family-bind/`：按当前 role 分叉 UI —— 老人端显示邀请码 + 复制；子女端输入框 + 兑换按钮
8. `pages/profile/`：菜单加「切换角色」和「家庭绑定」两项
9. `custom-tab-bar/index.js`：在 `attached` 时读 `App.globalData.role` 存到 data，为 C6 预留；视觉先不变

### 测试
- `tests/familyService.test.js`：`generateInviteCode` 格式校验；`redeemInviteCode` 的"自己兑换自己"拒绝路径；"兑换成功后双方写入"路径（mock 云 DB）

## 成功判据

1. 未选角色的用户打开小程序 → 跳转到 role-switch → 选了之后进 home，再次打开不再跳
2. 老人端 profile 页点「家庭绑定」→ 显示邀请码；子女端同路径 → 显示输入框
3. 子女端输入老人邀请码兑换成功后，双方 `user_profile.boundFamilyId` 都落库
4. `family_bindings` / `family_inbox` 两个集合创建成功（MCP readNoSqlDatabaseStructure 可见）
5. `npm test` 全绿，覆盖率门槛不破（statements ≥ 70%、functions ≥ 75%）
6. 老人端现有体验零回归：tabBar 视觉不变、健康/商城/AI/订单等流程如旧

## Docs Impact

- `docs/README.md`：集合表加 `family_bindings` / `family_inbox` 两行；`user_profile` 字段补 role/constitution/boundFamilyId；服务层表加 familyService / familyInboxService 两行
- `docs/plans/README.md`（如存在）或 `docs/README.md` plans 索引：追加本 sprint

## 任务分解

- **T1** — 云端建 `family_bindings` 与 `family_inbox` 集合（MCP）
- **T2** — `profileService.js` 扩字段 + 精简 catch + 新增 `setRole`
- **T3** — `familyService.js` + 单元测试（纯函数 `generateInviteCode` + `redeemInviteCode` mock 路径）
- **T4** — `familyInboxService.js`（读 + 标已读 + 写）
- **T5** — `pages/role-switch/`（js/json/wxml/wxss）
- **T6** — `pages/family-bind/`（js/json/wxml/wxss，按 role 分叉）
- **T7** — `app.js` 启动时读 role，空则跳 role-switch；`app.json` 注册新页
- **T8** — `pages/profile/` 加两个菜单项
- **T9** — `custom-tab-bar` 读 globalData.role 预留
- **T10** — `docs/README.md` 数据契约 + 索引更新
- **T11** — 跑 `npm test`，修到全绿，覆盖率不降
- **T12** — Phase 6 squash + merge + push

## 明确不做

- 子女端独立 tabBar（父母动态/补货/简报三个页面）→ **C6**
- 异常检测自动写 family_inbox → **C6**
- 体质字段填数据 → **C2**
- 订阅/问诊字段 → **C7**
- 老数据迁移 —— 用户说"无数据无兼容"，直接 schemaless 写入新字段

## 风险

- **邀请码碰撞**：6 位字母数字有 ~22 亿组合，demo 阶段不加唯一索引也够用；C6/正式上线前再加
- **re-launch 跳转循环**：role-switch 自己必须不能跳 role-switch，用 `@redirect-from=boot` 参数防护
- **custom-tab-bar 的 role 分叉**：本 sprint 不改视觉，仅把 role 读进 data。C6 再真正分叉 tabList
