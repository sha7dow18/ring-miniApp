# 系统架构 · 心络智医

> 作为首席架构师对这个小程序的完整描述。**任何人接手前必读**。
>
> 原则：**领域驱动**划子系统、**家庭子系统作为跨端总线**、**高内聚低耦合**。

## 2. 依赖关系（有向）

```
                      [Identity]
                    /      |      \
                   /       |       \
                  ▼        ▼        ▼
             [Health]─┬──▶[AI]◀──[Content]
                      │    ▲
                      │    │
                      └──▶ │(体质 + 健康上下文)
                           │
               [Commerce] ─┴─▶ 支付 hook ──▶ replenishment_plans
                                                    │
                                                    ▼
                                              [Family 总线]
                                                    │
               [Health 异常] ──────push────▶ family_inbox
               [Commerce 补货] ────push────▶ family_inbox
               [AI 周简报] ───────share────▶ weekly_digests
                                                    │
                                                    ▼
                                              子女端 UI
```

### 规则
- **Identity 是根依赖**：所有子系统消费 `profileService`（读 openid、role、constitution、boundFamilyId）
- **Family 是通信总线**：所有老人端 → 子女端的数据流只走两条路
  - 事件/通知：`family_inbox`
  - 结构化数据：`weekly_digests.sharedWith`
- **不允许**：子系统 A 直接读子系统 B 的私有集合。必须通过 B 的 service 层

## 3. 高内聚 / 低耦合 的实现

### 高内聚（做得好的地方）
| 模块 | 内聚体现 |
|---|---|
| `constitutionService` | 9 体质字典 + prompt 构建 + AI 解析 + 云持久 + 回写 profile，一处完整 |
| `replenishService` | 计划生成 + 到期判断 + 状态机，只面向补货域 |
| `digestService` | 汇总 + AI + 共享字段逻辑集中 |

### 低耦合（当前瑕疵 + 本次修复）

| 瑕疵 | 修复 |
|---|---|
| `pages/home` 直接 `wx.cloud.database().collection("family_bindings")` 查绑定的 childOpenId | 抽 `familyService.getBoundChildOpenId()`，home 和 digestService 都用它 |
| `digestService` 同样直接查 `family_bindings` | 同上 |
| `orderService.payOrder` require `replenishService` | 保留 — 这是业务顺承；更优是事件模型但 demo 不必过度工程化 |
| 硬编码的 role 字符串散落 `"elder"` / `"child"` | 集中在 `profileService.ROLES` 导出 |

### 关键抽象
- **`familyService.getBoundChildOpenId(): Promise<string | null>`** — 所有老人端想"发给我的子女"都走这一个函数
- **`familyInboxService.pushToInbox({ toOpenId, type, title, body, payload })`** — 所有跨端事件的唯一 sink
- **`replenishDispatcher.dispatchToChild(toOpenId): Promise<number>`** — 电商 → 家庭的适配层

## 4. Family 子系统详解

### 4.1 数据模型

```
┌─ user_profile (PRIVATE) ──────────┐
│  _openid, role, constitution,     │
│  boundFamilyId ─────────────────────────┐
└───────────────────────────────────┘     │
                                          ▼
┌─ family_bindings (CUSTOM) ────────────────────────┐
│  _id                                              │
│  _openid: elder                    (auto-filled)  │
│  childOpenId: child                (redeem 时写)   │
│  inviteCode: "ABC123"  6 位大写字母数字            │
│  status: "pending" | "bound"                      │
│  createdAt / boundAt                              │
└───────────────────────────────────────────────────┘

┌─ family_inbox (CUSTOM) ───────────────────────────┐
│  toOpenId: 收件人 openid（必填）                   │
│  type: "health_anomaly" | "replenish_due"         │
│       | "weekly_digest" | "daily_snapshot"        │
│       | "general"                                  │
│  title: string                                    │
│  body: string                                     │
│  payload: { ... 依 type 变化 ... }                 │
│  read: boolean                                    │
│  createdAt / readAt                               │
└───────────────────────────────────────────────────┘

┌─ weekly_digests (CUSTOM: _openid OR sharedWith) ──┐
│  _openid: elder                                   │
│  sharedWith: childOpenId (可读者)                  │
│  weekStart: "YYYY-MM-DD"                          │
│  summary: { avgSleepScore, avgHrResting, ... }    │
│  headline / highlights[] / concerns[]             │
│  recommendations[] / tone                         │
└───────────────────────────────────────────────────┘
```

### 4.2 权限模型（CloudBase 安全规则）

| 集合 | Read | Create | Update | Delete |
|---|---|---|---|---|
| `family_bindings` | auth.openid != null | auth.openid != null | auth.openid != null（兑换时 child 要能写） | doc._openid == auth.openid |
| `family_inbox` | doc.toOpenId == auth.openid | auth.openid != null | doc.toOpenId == auth.openid | doc.toOpenId == auth.openid |
| `weekly_digests` | doc._openid == auth.openid **OR** doc.sharedWith == auth.openid | auth.openid != null | doc._openid == auth.openid | doc._openid == auth.openid |

**生产收紧建议**（demo 阶段不做）：
- family_inbox.create 应限定 "发件人与收件人必须在同一 family_bindings"
- family_bindings.update 应限定 "仅 pending + 接收者是 childOpenId 本人"

### 4.3 跨端交互

```
老人端                         家庭总线                    子女端
────────                      ────────                   ────────
home onShow
  ├─ anomalyDetector ────▶ family_inbox      ─read─▶  family-home inbox
  │    (type=health_anomaly)
  │
  ├─ replenishDispatcher ──▶ family_inbox   ─read─▶  replenish tab
  │    (type=replenish_due, 去重)
  │
  └─ (可选)                                  ─read─▶  family-home summary

digest 页点"生成" 
  ├─ digestService.generateForMe
  │    ├─ 写 weekly_digests(sharedWith=child)  ─read─▶  digest 列表
  │    └─ push inbox(type=weekly_digest)       ─read─▶  inbox 新条

family-bind 页
  ├─ createPendingBinding ─▶ family_bindings 新行(status=pending)
  └─ 显示 inviteCode

                                               family-bind 页
                                                 ├─ redeemInviteCode ─▶ 查 & update
                                                 └─ 双方 setBoundFamilyId
```
